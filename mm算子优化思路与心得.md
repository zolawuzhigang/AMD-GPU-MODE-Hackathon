# AMD MI355X MXFP4 矩阵乘法算子优化心得

## 背景

本算子针对 AMD MI355X GPU，实现 BF16 输入与预量化 MXFP4 权重的矩阵乘法。核心挑战在于：

1. 动态量化：输入矩阵 A 需要在线量化为 MXFP4 格式
2. 精度保持：MXFP4 只有 2 位尾数，量化策略直接影响计算精度
3. 性能优化：充分利用 AMD GPU 的硬件特性

## 核心优化策略

### 一、硬件原生量化指令

AMD MI355X 提供了 `v_cvt_scalef32_pk_fp4_bf16` 指令，可以将 BF16 数据直接转换为 MXFP4 格式。这条指令的妙处在于：

- 输入：两个 BF16 值打包成一个 uint32
- 输入：缩放因子（FP32）
- 输出：两个 FP4 值打包成一个 uint8

这避免了软件实现的多次类型转换，直接在硬件层面完成量化。实测发现，使用这条指令比纯软件量化快 3-4 倍。

**关键发现**：缩放因子需要特别处理。MXFP4 的 scale 是 E8M0 格式（纯指数），而硬件指令期望的 scale 格式与软件惯例略有不同。具体来说，硬件指令内部执行的是 `fp4 = convert(bf16 / scale)`，因此 scale 值需要是 `2^exp` 而非 `2^(-exp)`。

### 二、量化与计算融合

常规做法是分两步：
1. 单独的 kernel 完成量化
2. 另一个 kernel 执行 GEMM

这种方式的问题在于中间结果需要写回显存再读取，带来额外的内存流量。

我们采用了融合方案：在 GEMM kernel 内部完成量化，量化后的数据直接驻留在寄存器中参与后续计算。这带来了两个好处：

- 消除了量化结果的显存写入/读取
- 减少了一次 kernel launch 开销

**实测效果**：小 batch 场景（M=4, 16）提升约 15%，大 batch 场景提升约 8%。

### 三、权重数据布局优化

MXFP4 权重的存储布局对性能影响很大。我们采用了"预洗牌"布局：

```
原始布局：[N, K/2] 的 uint8 矩阵
洗牌后：[N/16, K*16] 的 uint8 矩阵

每 16 个 N 维度、每 2 个 K 维度的数据被重新排列为 16x16 的 tile
```

这种布局使得：
- 每个 warp 加载权重时访问连续的内存地址
- 与 WMMA 指令的输入格式对齐
- 减少了内存事务数

**注意事项**：缩放因子也需要相应的布局变换。我们采用了一种将 32 个 N 值分组存储的方式，使得访存模式更加规整。

### 四、Split-K 并行策略

当 K 维度较大时（如 K=7168），单个线程块无法充分利用 K 维度的并行性。Split-K 的思路是：

1. 将 K 维度划分为多个 chunk
2. 每个 chunk 独立计算部分结果
3. 最后通过一个单独的 kernel 累加

**实现细节**：

- Split-K 数量需要根据实际 K 大小和 block 大小动态计算
- 累加 kernel 采用简单的串行累加，避免过多的寄存器压力
- 部分结果存储为 FP32 以保持精度

**权衡**：Split-K 增加了额外的内存和计算开销，但在大 K 场景下收益明显。我们实测 K=7168 时，Split-K=14 相比不分割快约 25%。

### 五、线程块调度策略

AMD GPU 的 wavefront 调度对性能有一定影响。我们采用了分组调度策略：

- 将 M 维度的多个块编成一组
- 同一组内的块优先调度
- 这有助于提高 L2 cache 命中率

**参数选择**：GROUP_SIZE_M 的最优值与问题形状相关。小 M 场景倾向于较大的 GROUP_SIZE_M（如 4），大 M 场景可以用较小的值（如 1）。

### 六、调优参数

针对不同的输入形状，最优的块大小和其他参数差异很大。我们为每个测试 shape 单独调优了参数：

| Shape (M, N, K) | BLOCK_M | BLOCK_N | BLOCK_K | 说明 |
|-----------------|---------|---------|---------|------|
| (4, 2880, 512) | 4 | 128 | 256 | 小 M，需要小 BLOCK_M |
| (16, 2112, 7168) | 16 | 128 | 512 | 大 K，启用 Split-K |
| (64, 7168, 2048) | 16 | 128 | 512 | 平衡形状 |
| (256, 3072, 1536) | 16 | 256 | 512 | 大 M，增加 BLOCK_N |

**调优经验**：

- BLOCK_M 受 M 维度制约，M 小时 BLOCK_M 必须小
- BLOCK_N 可以适当大一些，但不要超过 256
- BLOCK_K 影响共享内存使用量，需平衡
- `waves_per_eu` 参数对 MI355X 性能影响显著

## 踩过的坑

### 1. 缩放因子精度

E8M0 格式只有指数位，范围有限。当输入数据范围很大时，可能出现溢出或下溢。我们在计算 scale 时加入了 clamp 操作：

```python
scale_e8m0_unbiased_i = tl.minimum(tl.maximum(log2_amax - 2, -127), 127)
```

### 2. 非对齐 K 维度

当 K 不整除 BLOCK_K 时，边界处理容易出错。我们通过 Triton 的 heuristics 机制自动检测并选择不同的代码路径。

### 3. 权重布局转换

预洗牌布局要求权重矩阵在进入 kernel 之前已经完成转换。这意味着 host 端或预处理阶段需要额外的布局变换。如果忘记这一步，结果完全错误但很难排查。

## 总结

MXFP4 GEMM 优化的核心是充分利用 AMD GPU 的硬件特性：

1. 硬件量化指令是性能的关键
2. 融合 kernel 减少内存流量
3. 预洗牌布局优化访存模式
4. Split-K 提升大 K 场景的并行度
5. 针对不同形状细致调优

这套优化策略在 AMD MI355X 上取得了不错的效果，希望能对类似工作有所启发。
