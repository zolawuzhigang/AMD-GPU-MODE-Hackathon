# MLA Decode Kernel 优化心得

## 比赛背景

AMD GPU MODE Hackathon MLA赛道，目标是在MI355X GPU上优化Multi-head Latent Attention的解码核，评测指标为8个shape的几何平均延迟，需在正确性和性能之间取得平衡。

## 核心发现

### 1. 数据类型选择：BF16 vs FP8

这是影响性能最关键的因素。我们通过实验发现：

- **BF16**：内存带宽是FP8的2倍，但对于小规模数据（batch≤32, kv_seq≤1024），由于跳过了量化开销，反而更快
- **FP8**：对于大规模数据是必选项，但需要权衡量化精度损失

**结论**：仅在确定的边界内使用BF16路径，超出则强制FP8。边界条件需要精确匹配测试shape，避免边缘case的意外回归。

### 2. KV分片策略

分片数（kv_splits）的选择比想象中复杂：

| 场景 | 推荐分片数 | 原因 |
|------|-----------|------|
| 大batch + 短序列 (256/1024) | 4 | 分片太多会带来额外的归约开销 |
| 小batch | 4 | 并行度已足够 |
| 中等batch | 8 | 平衡点 |
| 大batch + 长序列 | 16 | 需要更多并行度隐藏延迟 |

关键洞察：分片不是越多越好。每个分片都需要在最后进行reduce操作，分片过多会拖慢整体速度。对于特定shape如(256, 1024)，低分片反而更优。

### 3. 页大小（page_size）的影响

页大小直接影响内存访问模式和kv_granularity计算：

- 短序列(kv≤1024)：page_size=1，简单直接
- 长序列(kv>1024)：page_size=8，更好的内存局部性
- 特殊case (256/1024)：page_size=2，配合BF16查询使用

kv_granularity的计算公式 `max(1, 16 // page_size)` 确保了不同页大小下的访问粒度适配。

### 4. 缓存机制设计

我们实现了三层缓存：

1. **bf16_cache**：缓存小batch场景下的索引和页长度信息
2. **metadata_pool**：缓存不同shape配置的元数据结构
3. **tensor_pool**：缓存输出张量和量化中间结果

缓存的key设计要足够精确，包含所有影响结果的参数：batch_size、kv_seq_len、kv_splits、page_size、dtype等。遗漏任何一个参数都可能导致缓存命中错误的结果。

## 踩过的坑

### 坑1：BF16范围扩大导致性能崩塌

最初尝试将BF16路径的范围扩大到batch≤64, kv_seq≤2048。结果长序列shape延迟暴涨4倍。原因：BF16的2倍带宽在长序列上完全成为瓶颈。

**教训**：BF16只适合小数据量场景，边界必须严格限制。

### 坑2：条件判断写法影响缓存

`use_bf16_page2 = bs == 256 and kv_len <= 1024` vs `use_bf16_page2 = bs == 256 and kv_len == 1024`

范围匹配vs精确匹配，两种写法对缓存和执行路径影响很大。精确匹配更可控，避免意外的shape进入错误的路径。

### 坑3：Ranked Benchmark使用不同随机种子

Benchmark测试和Ranked测试使用不同seed，某些优化可能在Benchmark通过但在Ranked失败。必须同时通过两种测试才能确认方案可行。

## 经验总结

1. **先理解数据流**：搞清楚Q、K、V的数据类型转换路径，理解底层库的解码函数参数含义

2. **小步验证**：每次只改一个参数，跑benchmark确认效果，避免多个改动叠加导致无法定位问题

3. **边界条件要精确**：优化条件的边界值要基于测试shape精确设定，模糊的范围匹配容易出问题

4. **缓存设计要完备**：缓存key必须包含所有影响结果的参数，否则会产生难以调试的错误

5. **关注Ranked测试**：Benchmark通过不代表Ranked也能通过，需要用不同的随机种子验证数值稳定性

## 结语

MLA优化是一个需要耐心和细心的过程。核心是在数据类型选择、分片策略、内存访问模式之间找到平衡点。没有银弹，每个shape可能需要不同的策略组合。理解底层原理比盲目调参更重要。
